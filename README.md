# Project Sentinel Loop

## 1. Project Overview

**Project Sentinel Loop** is a production-grade, closed-loop SOAR (Security Orchestration, Automation, and Response) pipeline engineered to orchestrate real-time defensive workflows. By bridging enterprise endpoint data layers via **LimaCharlie** with a stateful **n8n** automation architecture and a locally throttled **Google Gemini AI** engine, Project Sentinel Loop mimics a natural immune response to modern endpoint attacks.

* **Value Proposition:** Reduces standard Mean Time to Detect (MTTD) and Mean Time to Respond (MTTR) from hours to isolated machine containment in less than 60 seconds.

---

## 2. System Architecture & Workflow Loop

The defensive platform executes deterministically via a five-staged architecture loop:

1. **Ingestion:** A Slack webhook triggers on high-criticality telemetry alerts sent from a LimaCharlie endpoint detector.
2. **Forensic Pull:** The pipeline automatically contacts LimaCharlie's storage layer to extract raw file artifacts or compressed packages.
3. **Static AI Engineering:** An advanced n8n AI agent analyzes basic strings and programmatically instructs `gemini-3.5-flash` to author a structurally sound, production-ready YARA rule tailored precisely to the suspicious payload indicators.
4. **Global Deployment:** The freshly generated signature is pushed upstream directly into LimaCharlie's centralized Config Hive structure and instantly set to an active monitoring status.
5. **Autonomous Fleet Containment:** A permanent, regex-based master Detection & Response (D&R) rule matches the signature rollout across the fleet, instructing the local kernel extension to apply network containment (`history_isolate`) instantly on any positive matches.

---

## 3. End-to-End Data Flow & n8n Logic

When an attack triggers a custom D&R rule (such as the **Ransomware** detection engine):

1. The malicious file artifact is safely collected inside LimaCharlie.
2. An automated notification containing the unique `Artifact ID` and `Organization ID (OID)` is dispatched via webhook to n8n.
3. Inside the n8n canvas, an **HTTP Request node** pulls down the raw artifact content.
4. A JavaScript **Code node** cleans and parses important metadata/strings from the artifact, passing a structured payload to the **AI Agent**.
5. The AI Agent dynamically designs a custom YARA signature to fingerprint the threat.
6. A final **HTTP Request node (Form-Urlencoded)** sends the active rule back to LimaCharlie's Config Hive, closing the loop.

## 4. Operational Testing & Validation

### Simulated Attack Vector

To validate the automated pipeline, execute the following commands in the console of a target LimaCharlie sensor. This script drops a test file simulating a **"Living off the Land" Ransomware Attack**—frequently used by threat groups such as *ShrinkLocker*, *Lorenz*, and *DEV-0270 (Phosphorus)* to abuse native administrative encryption utilities.

```bash
# Command 1: Stage the payload inside the temp directory
run --shell-command "echo X5O!P%@AP[4\PX5O!P)7CC)7}$EICAR-STANDARD-ANTIVIRUS-TEST-FILE!$H+H* > C:\Windows\Temp\eicar_ransomware.bat && echo reg add HKLM\SOFTWARE\Policies\Microsoft\FVE /v EnableBDEWithNoTPM /t REG_DWORD /d 1 /f >> C:\Windows\Temp\eicar_ransomware.bat"

# Command 2: Execute the simulated ransomware routine
run --shell-command "C:\Windows\Temp\eicar_ransomware.bat"

```

### Detection Logic

* **Why it triggers:** Modifying the registry path `HKLM\SOFTWARE\Policies\Microsoft\FVE` with the value `EnableBDEWithNoTPM` is a high-confidence indicator of an environment trying to force full disk BitLocker encryption while bypassing physical hardware requirements (TPM chips)—allowing attackers to seamlessly hijack native tools, lock systems, and evade traditional signature-based binaries.

## 5. Deployment Custom D&R Rules

### Rule 1: Ransomware (The Initial Trigger)

Monitors for the telltale signs of native BitLocker policy manipulation or unexpected drop folders (`C:\Windows\Temp\*.bat`) to alert the n8n endpoint and pull the diagnostic file.

### Rule 2: Dynamic-YARA (The Master Containment Hook)

A regex-based permanent master engine that dynamically evaluates incoming fleet telemetry. If any device matches a YARA rule prefix generated by the pipeline (e.g., `^my-rule\d+`), it executes the `history_isolate` task, instantly neutralizing lateral threat spread across the network.
